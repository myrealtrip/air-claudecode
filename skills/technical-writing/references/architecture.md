# Document Architecture (구조)

> Source: [toss/technical-writing - architecture](https://github.com/toss/technical-writing)

## 6 Principles

### 1. One Thing Per Page (한 페이지에 하나만)

하나의 문서는 하나의 구체적 주제만 다룬다.

- H3(###)까지만 사용. H4(####)가 필요하면 문서를 분리한다
- 포괄적 주제는 overview 페이지로 묶고 하위 문서로 분리한다
- 두 개 이상의 핵심 주제가 있으면 문서를 나눈다

| Bad | Good |
|-----|------|
| `# React 사용법` → `## 컴포넌트` → `### 기본 구조` → `#### JSX 사용법` → `##### 세부사항` | `# React 컴포넌트 생성하기` → `## 기본 구조` → `## 상태 관리` → `## 라이프사이클` |

**Why**: 가독성이 높아지고, 탐색이 쉬워지며, 유지보수가 편해진다.

### 2. Value-First (가치 먼저)

독자가 얻을 수 있는 가치를 먼저 설명한다. 기능 중심의 설명을 피한다.

- 배경 설명이나 부가 정보는 뒤로 미룬다
- "이 문서를 읽으면 무엇을 할 수 있는가?"를 먼저 전달한다
- 기능 나열보다 독자가 얻는 이점을 강조한다

| Bad | Good |
|-----|------|
| 리버스 프록시 설정은 2019년에 도입되었고... | 리버스 프록시 설정을 적용하면 네트워크 지연 문제를 최소화할 수 있어요 |
| 이 스니펫은 다양한 설정 옵션을 제공합니다. 먼저 `connection_timeout`... | 이 스니펫을 사용하면 PostgreSQL 데이터베이스 연결 속도가 50% 빨라집니다 |

### 3. Effective Headings (효과적인 제목)

제목은 문서의 핵심 내용을 간결하게 전달해야 한다.

- 핵심 키워드를 포함한다
- 30자 이내로 작성한다
- 평서문으로 작성한다 (의문문, 느낌표 금지)
- 문서 전체에서 일관된 스타일을 유지한다 (동사형/명사형 통일)
- 단계가 있다면 숫자를 사용한다

| Bad | Good |
|-----|------|
| `# 에러를 해결하는 방법은?` | `# NOT_FOUND_USER 에러를 해결하는 방법` |
| `## 키워드를 포함하세요` / `## 일관성 유지` / `## 평서문으로 작성하기` | `## 키워드 포함하기` / `## 일관성 유지하기` / `## 평서문으로 작성하기` |

### 4. Essential Overview (핵심 개요)

모든 문서는 제목 바로 아래에 핵심 개요를 포함한다.

- 문서 전체를 읽지 않아도 개요만으로 주제를 파악할 수 있어야 한다
- "이 문서를 읽으면 무엇을 할 수 있는가?"에 대한 답을 제공한다
- 기술적 배경보다 핵심 정보를 우선한다

| Bad | Good |
|-----|------|
| TypeScript의 유틸리티 타입을 소개합니다. Partial, Pick, Omit 등을 사용할 수 있어요. | TypeScript의 유틸리티 타입을 사용해서 객체 타입을 변형하는 방법을 알아볼게요. Partial, Pick, Omit 등을 사용하여 반복적인 타입 선언을 줄이고 유연하게 타입을 관리할 수 있어요. |

### 5. Predictable Structure (예측 가능한 구조)

문서는 일관된 구조를 가져야 한다. 독자의 인지 부담을 줄인다.

- 일관된 제목 패턴을 유지한다 (같은 수준의 소제목 = 같은 문체)
- 논리적 순서로 정보를 배치한다 (기본 개념 → 사용법 → 예제 → 심화)
- 용어를 통일한다 (하나의 개념 = 하나의 용어)
- 설명 → 코드 순서로 배치한다 (코드 먼저 나오면 독자가 해석해야 함)

| Bad | Good |
|-----|------|
| `## 비동기 데이터 요청하기` → `## 기본적인 사용법` | `## 기본적인 사용법` → `## 비동기 데이터 요청하기` |
| "상태(state)", "데이터 저장", "값 유지" 혼용 | "상태(state)"로 통일 |

### 6. Detailed Explanations (상세한 설명)

독자의 이해 수준에 맞게 충분한 정보를 제공한다.

- 새로운 개념은 등장 시점에 한두 문장으로 정의한다
- 기능의 동작 조건, 단위, 입력/출력을 구체적으로 명시한다
- 다른 문서에서 참조되는 값은 출처와 사용처를 연결한다

| Bad | Good |
|-----|------|
| 이 서비스는 이벤트 소싱 방식을 사용해 상태를 관리합니다. | 이 서비스는 이벤트 소싱(Event Sourcing) 방식을 사용해 상태를 관리합니다. 이벤트 소싱은 상태의 최종 결과만 저장하는 대신, 상태 변화를 일으킨 모든 이벤트를 기록하는 방식입니다. |
| `sessions[].duration`: 세션의 지속 시간 | `sessions[].duration`: 세션의 지속 시간(밀리초). 수동 로그아웃이면 실제 이용 시간, 시간 초과면 마지막 활동 시점까지의 시간 |

## Outline Template

```markdown
# {문서 제목}                    <- H1: 하나만

{개요: 이 문서가 무엇을, 누구를 위해, 왜}

## {핵심 섹션 1}                 <- H2: 가치 순서대로
### {하위 주제}                  <- H3: 최대 깊이

## {핵심 섹션 2}

## 다음 단계 / 참고              <- 부가 정보는 마지막
```
