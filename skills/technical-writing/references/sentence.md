# Sentence Refinement (문장 다듬기)

> Source: [toss/technical-writing - sentence](https://github.com/toss/technical-writing)

## 5 Rules

### 1. Clarify the Subject (주어 명확히)

문서를 읽는 개발자가 곧 행동의 주체이다. 도구나 시스템을 행위 주체로 쓰지 않는다.

**규칙**:
- 도구/기술/시스템을 주어로 사용하지 않는다 (단, 시스템 동작 자체를 설명할 때는 예외)
- 능동형을 사용한다. 수동형 문장은 주체가 불분명해진다

| Bad | Good |
|-----|------|
| **이 라이브러리는** 데이터베이스 초기화를 수행해요 | 이 명령어를 실행하면 데이터베이스를 초기화할 수 있어요 |
| **이 코드는** API 요청으로 인증 토큰을 추가해요 | 이 코드를 사용하면, API를 요청했을 때 자동으로 인증 토큰을 추가할 수 있어요 |
| 애플리케이션이 실행되기 전에 설정이 완료**되어야 합니다** | 애플리케이션을 실행하기 전에 설정을 완료**하세요** |
| 변경 사항이 적용된 후 다시 빌드**되어야 합니다** | 변경 사항을 적용한 후 다시 빌드**하세요** |

**예외**: 시스템 동작 자체를 설명할 때는 도구가 주어가 될 수 있다.
```
Impression 로그는 두 가지 방식으로 동작해요.
1. 특정 접근성 요소가 화면에 나타나면 자동으로 수집돼요.
2. 특정 DOM 요소를 직접 지정해서 뷰포트에 나타날 때 로그를 수집할 수 있어요.
```

### 2. Essential Information Only (필수 정보만)

짧은 문장을 쓴다. 한 문장에 하나의 생각만 담는다.

**규칙**:
- 문장이 길어지면 두 문장으로 나눈다
- 메타 담화를 최소화한다: '앞서 설명했듯이', '이제 알아보겠습니다', '아시겠지만' 등 제거
- 불필요한 수식어를 제거한다

| Bad | Good |
|-----|------|
| 버튼을 클릭하면 다음 단계로 이동하게 되며, 그 이후의 작업을 진행할 수 있습니다 | 버튼을 클릭하면 다음 단계로 이동합니다. 이동 후 이어서 작업을 진행하세요 |
| 이 API를 호출할 때 요청 헤더를 포함해야 하며, 올바른 인증 정보를 제공해야 정상적으로 응답을 받을 수 있습니다 | 이 API를 호출할 때 요청 헤더와 인증 정보를 포함하세요 |
| 앞에서 설명했지만, 결론은 이 설정을 변경하는 것이 가장 효과적인 방법이라는 겁니다 | 이 설정을 변경하는 것이 가장 효과적인 방법입니다 |
| 아시다시피, 이 설정은 성능에 영향을 줍니다 | 이 설정은 성능에 영향을 줍니다 |

**메타 담화(Metadiscourse)**: 핵심 메시지 자체가 아닌 보조 표현
1. 문서 흐름 설명: '다음으로', '앞서 설명했듯이', '이제 알아보겠습니다'
2. 불필요한 강조: '사실은', '솔직히 말하면', '결론적으로'
3. 독자 반응 유도: '아시겠지만', '여러분도 아실 것입니다'

### 3. Write Concretely (구체적으로)

명사형 표현보다 동사를 사용한다. 모호한 표현을 구체적으로 바꾼다.

**규칙**:
- 동사에서 파생된 명사('수행', '진행', '처리')를 동사로 되돌린다
- '가능성이 있다', '일부 경우' 같은 모호한 표현을 명확하게 한다
- 누가, 무엇을, 어디에, 어떻게 하는지 명시한다
- 구체적인 수치나 기준을 제시한다
- 참조되는 값의 출처와 사용처를 연결한다

| Bad | Good |
|-----|------|
| 코드 최적화 진행 후 배포 수행이 필요합니다 | 코드를 최적화한 후 배포하세요 |
| MongoDB 연결 정보 설정 및 초기화가 필요합니다 | MongoDB에 연결할 호스트와 포트를 설정하고 데이터베이스를 초기화합니다 |
| 설정 파일을 변경하면 기존 설정이 영향을 받을 수도 있습니다 | 설정 파일을 변경하면 기존 설정이 삭제됩니다 |
| 파일을 업로드하면 자동으로 저장됩니다 | 사용자가 파일을 업로드하면 서버에 자동으로 저장됩니다 |
| 에러가 발생하면 로그를 확인하세요 | 에러가 발생하면 `/var/log/app/error.log`를 확인하세요 |
| 데이터가 많을 때는 성능이 저하될 수 있습니다 | 데이터가 10,000건을 넘으면 응답 시간이 1초 이상 걸립니다 |
| 에러가 발생하면 Exception을 던집니다 | 에러가 발생하면 예외(`Exception`)를 일으킵니다 |

### 4. Natural Korean (자연스러운 한국어)

불필요한 한자어를 제거한다. 번역체 표현을 자연스러운 한국어로 바꾼다.

**규칙**:
- '수행하다', '실행하다', '진행하다' 같은 단어는 빼도 의미가 전달되면 제거한다
- 영어식 명사형 표현(-tion, -ing)을 한국어 동사로 바꾼다

| Bad | Good |
|-----|------|
| 로그 파일을 삭제하는 작업을 수행합니다 | 로그 파일을 삭제합니다 |
| 설정 변경을 수행한 후 적용을 진행합니다 | 설정을 바꾼 뒤 적용합니다 |
| API 키를 이용한 사용자 인증 처리가 완료된 후, 데이터베이스 접속 설정 진행이 가능합니다 | API 키로 사용자를 인증한 후, 데이터베이스에 접속하도록 설정할 수 있습니다 |
| 시스템 모니터링 수행을 통해 서버 성능 측정 작업을 실시하게 됩니다 | 시스템을 모니터링해서 서버 성능을 측정합니다 |
| 보안 정책 업데이트 진행 후 시스템 재시작 처리를 수행해야 합니다 | 보안 정책을 업데이트한 후 시스템을 재시작합니다 |

### 5. Maintain Consistency (일관성 유지)

공식 기술 용어를 따른다. 같은 개념에 여러 표현을 사용하지 않는다.

**규칙**:
- 기술 용어는 위키피디아 및 공식 문서의 표기를 따른다 (대소문자 주의)
- 하나의 문서 안에서 같은 개념을 다른 단어로 표현하지 않는다
- 약어는 처음 사용 시 전체 이름을 괄호로 병기한다. 약어와 괄호 사이는 붙여 쓴다
- 외래어 표기는 사용 빈도를 고려한다 (구글 트렌드 기준 5배 이상이면 관용 표기)

| Bad | Good |
|-----|------|
| K8을 사용하면 배포가 쉬워집니다 | 쿠버네티스(Kubernetes)를 사용하면 배포가 쉬워집니다 |
| 파일을 추가/첨부/넣다 혼용 | "업로드"로 통일 |
| 이 기능은 SSR을 지원합니다 | 이 기능은 SSR(Server-Side Rendering)을 지원합니다 |
| 캐시/캐싱/Caching 혼용 | "캐시"로 통일 |
| 프런트엔드 (맞춤법) | 프론트엔드 (관용 표기, 5배 이상 사용) |
